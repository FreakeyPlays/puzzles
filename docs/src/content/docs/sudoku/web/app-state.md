---
title: App State Machine
description: States, transitions and persistence model for the sudoku-web application
---

The app is modelled as a finite state machine. At any point in time the app is in exactly one of four states. Two of them are runtime-only; two are persisted to LocalStorage so the correct UI is restored on the next app open.

## States

| State     | Persisted | Timer   | Meaning                                         |
| --------- | --------- | ------- | ----------------------------------------------- |
| `idle`    | ✅        | —       | No active game. Home shows **Start Game**.      |
| `loading` | ❌        | —       | A puzzle is being generated by the WASM worker. |
| `playing` | ✅        | running | A puzzle is active and the user is playing.     |
| `paused`  | ✅        | stopped | A puzzle is active but the timer is stopped.    |

`loading` is never written to LocalStorage. If the app is killed during puzzle generation, no puzzle was persisted yet — the next open returns to `idle`.

## State Diagram

```
                  ┌─────────────────────────────────────────┐
                  │         Start Game + Difficulty          │
                  ▼                                          │
 ┌────────┐    ┌─────────┐    ┌─────────┐              ┌────┴───┐
 │  idle  │───►│ loading │───►│ playing │◄─────────────│ paused │
 └────────┘    └─────────┘    └────┬────┘              └────────┘
      ▲                            │                        ▲
      │                            │ visibilitychange       │
      │         Result Overlay     │ hidden / Tab switch    │
      │◄──[Home]────────────────── │ ──────────────────────►│
                                   │
                                   │ New Game (abandoned)
                                   ▼
                               loading ──► playing
```

## Transitions

| From      | To        | Trigger                                                                  |
| --------- | --------- | ------------------------------------------------------------------------ |
| `idle`    | `loading` | User selects difficulty and presses **Start Game**                       |
| `loading` | `playing` | `generate()` resolves — puzzle ready                                     |
| `playing` | `paused`  | `visibilitychange → hidden` (app backgrounded)                           |
| `playing` | `paused`  | User taps Home / History / Settings tab                                  |
| `paused`  | `playing` | `visibilitychange → visible` (app returns to foreground) — **automatic** |
| `paused`  | `playing` | User presses **Continue** on Home Screen                                 |
| `playing` | `loading` | User presses **New Game** — current puzzle marked `abandoned`            |
| `paused`  | `loading` | User presses **New Game** — current puzzle marked `abandoned`            |
| `playing` | `idle`    | User presses **Home** on the Result Overlay — puzzle marked `solved`     |

## Timer Behaviour

The timer maps directly onto the `playing` / `paused` distinction:

| Event                                              | Timer                                           |
| -------------------------------------------------- | ----------------------------------------------- |
| Enter `playing` (Start Game or Continue pressed)   | **Starts immediately**                          |
| `visibilitychange → hidden`                        | **Stops immediately**                           |
| `visibilitychange → visible` (still on `/game`)    | **Resumes immediately** — no user action needed |
| Continue pressed (after tab switch or app restart) | **Starts immediately**                          |

The elapsed time is stored in `sudoku:puzzle.elapsedSeconds` and updated on every pause. On app restart the timer picks up from the stored value.

## Puzzle Solved Detection

After every cell entry the app checks whether all 81 cells are filled:

```typescript
if (!currentBoard.includes("0") && currentBoard === solution) {
  // → puzzle solved
}
```

The `solution` string is kept in memory only (never persisted). It is returned by `generate()` when a new puzzle starts and recomputed via `solve(puzzle)` on every app restart. Comparing directly against the solution string avoids calling `validate()` for win detection — `validate()` is used separately for conflict highlighting.

## Win and Abandon Flow

**Win:** When `currentBoard === solution`, the Result Overlay appears on top of `/game`. The puzzle status is set to `solved` when the user leaves the overlay via **New Game** or **Home**.

**Abandon:** Pressing **New Game** during an active game presents two clearly distinct buttons — one to continue the current game (primary) and one to start a new game (secondary/destructive). If the user confirms, the current puzzle is marked `abandoned` and the App State transitions to `loading`.

There is no explicit "Give Up" button anywhere in the UI.

## Persistence Summary

| Key                | Stored phases                         |
| ------------------ | ------------------------------------- |
| `sudoku:app.phase` | `'idle'` \| `'playing'` \| `'paused'` |

`loading` is intentionally excluded. The `playing` / `paused` split means the app always knows whether the timer was running when the user last left — useful for future UX decisions around resume behaviour.
